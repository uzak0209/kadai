Hono とは？

超高速・軽量なWebフレームワーク

Node.js、Deno、Cloudflare Workers など様々な環境で動く

Hono の特徴

  • シンプル
    最小限のコードでAPIが作れる
  • 高速
    パフォーマンスが非常に良い
  • TypeScript完全対応
    型安全にAPIを開発できる
  • 軽量
    依存関係が少なく、バンドルサイズが小さい

Hono のコード例

import { Hono } from 'hono'

const app = new Hono()

// GET /api/todos
app.get('/api/todos', async (c) => {
  const todos = await db.query('SELECT * FROM todos')
  return c.json(todos)
})

// POST /api/todos
app.post('/api/todos', async (c) => {
  const body = await c.req.json()
  const todo = await db.insert('todos', body)
  return c.json(todo, 201)
})

export default app

シンプルで読みやすい！

JWT認証とは？

JSON Web Token

ユーザーを安全に認証するための仕組み

JWT の仕組み

トークン = 身分証明書のようなもの

一度ログインすれば、そのトークンを使って
「私は〇〇さんです」と証明できる

JWT認証フロー

sequenceDiagram
    participant U as ユーザー
    participant F as フロントエンド
    participant B as バックエンド
    participant D as DB

    U->>F: メール・パスワードを入力
    F->>B: POST /api/auth/login
    B->>D: ユーザー確認
    D->>B: ユーザー情報
    B->>B: JWTトークン生成
    B->>F: トークンを返す
    F->>F: トークンを保存
(localStorage)

    Note over F,B: 以降のリクエスト

    F->>B: GET /api/todos
(Header: Authorization: Bearer トークン)
    B->>B: トークン検証
    B->>D: データ取得
    D->>B: データ
    B->>F: レスポンス

JWT トークンの中身

eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.
eyJzdWIiOjEsImVtYWlsIjoidGVzdEBleGFtcGxlLmNvbSJ9.
vz0uGDqwoIf2zxpm7pUGRclV2yOeYQnWp5PJaYzJKqg

3つの部分に分かれている:

  • ヘッダー: アルゴリズム情報
  • ペイロード: ユーザー情報(ID、メールなど)
  • 署名: 改ざん防止用

JWT のメリット

  • ステートレス
    サーバーでセッション情報を持たなくていい
  • スケーラブル
    複数のサーバーで簡単に認証できる
  • 安全
    署名により改ざんを検出できる

TypeScript とは？

JavaScriptに型を追加した言語

Microsoft が開発

JavaScript の問題点

// JavaScriptの例
function add(a, b) {
  return a + b;
}

add(1, 2);        // 3 OK
add("1", "2");    // "12" えっ？文字列になる
add(1, "2");      // "12" バグの原因に...

型がないので、実行するまでエラーに気づけない

TypeScript の解決策

// TypeScriptの例
function add(a: number, b: number): number {
  return a + b;
}

add(1, 2);        // 3 OK
add("1", "2");    // エラー！コンパイル時に検出
add(1, "2");      // エラー！コンパイル時に検出

型を指定することで、バグを事前に防げる

TypeScript の特徴

  • 型安全
    コンパイル時にエラーを検出
  • エディタサポート
    自動補完・リファクタリング
  • ドキュメント代わり
    型定義がドキュメントになる
  • JavaScript互換
    既存のJSコードも使える

型定義の例

// Todoの型定義
type Todo = {
  id: number;
  title: string;
  description: string | null;
  completed: boolean;
  user_id: number;
  created_at: string;
  updated_at: string;
};

// 使い方
const todo: Todo = {
  id: 1,
  title: "牛乳を買う",
  description: null,
  completed: false,
  user_id: 1,
  created_at: "2024-01-01",
  updated_at: "2024-01-01"
};

Issue とは？

タスクやバグを管理する仕組み

Issueで管理するもの:

  • 新機能の提案
  • バグ報告
  • 改善案
  • 質問・議論

Issue の作成

# GitHubのWebUIで作成、または
gh issue create --title "ToDoの削除機能を追加" \
  --body "削除ボタンをクリックしたらToDoを削除できるようにする"

Issueには番号が振られる (例: #1, #2, #3...)

Issue の例

Issue #12: ToDoの削除機能を追加

説明:

現在、ToDoを削除する機能がありません。
ユーザーがToDoを削除できるようにしたいです。

実装内容:

  • 削除ボタンの追加
  • DELETE /api/todos/:id のAPI呼び出し
  • 削除後に一覧を更新

ブランチを切る

新しい機能は新しいブランチで開発

# mainブランチを最新にする
git checkout main
git pull

# 新しいブランチを作成して切り替え
git checkout -b feature/delete-todo

# ブランチ名の例:
# - feature/機能名 (新機能)
# - fix/バグ名 (バグ修正)
# - docs/ドキュメント名 (ドキュメント)

コードを編集してコミット

# ファイルを編集...

# 変更を確認
git status

# 変更をステージング
git add app/todos/page.tsx

# コミット (Issueの番号を含める)
git commit -m "feat: ToDoの削除機能を追加 #12"

# リモートにプッシュ
git push origin feature/delete-todo

Pull Request (PR) とは？

コードをレビューしてもらうための仕組み

PRの目的:

  • コードレビューを依頼
  • 変更内容を共有
  • テストの実行
  • mainブランチへの統合準備

PR の作成

# GitHubのWebUIで作成、または
gh pr create \
  --title "ToDoの削除機能を追加" \
  --body "Close #12

## 変更内容
- 削除ボタンを追加
- DELETE APIを呼び出す処理を実装
- 削除後に一覧を更新

## テスト方法
1. ToDoを作成
2. 削除ボタンをクリック
3. ToDoが削除されることを確認"

Close #12 でIssueと紐づけられる

PR の構成

📝 Pull Request #15: ToDoの削除機能を追加

  • タイトル・説明: 何を変更したか
  • 変更したコード (Diff): 追加・削除された行
  • コミット履歴: どう実装したか
  • レビュアー: 誰に見てもらうか
  • 関連Issue: Close #12
  • テスト結果: CI/CDで自動テスト

コードレビュー

チームメンバーがコードを確認

レビュアー

  • コードを読む
  • バグがないか確認
  • 改善点を提案
  • Approve or Request Changes

作成者

  • コメントに返信
  • 指摘を修正
  • 追加コミット
  • 再レビュー依頼

レビューコメントの例

レビュアー:

削除する前に確認ダイアログを表示した方が
ユーザーにとって親切だと思います。
window.confirm() を使ってはどうでしょうか？

作成者:

ご指摘ありがとうございます！
確認ダイアログを追加しました。

レビュー承認後

# レビュー承認されたらマージ
# GitHubのWebUIで "Merge pull request" をクリック

# または、コマンドで
gh pr merge 15 --merge

# マージ後、ブランチを削除
git checkout main
git pull
git branch -d feature/delete-todo

PRがマージされると、関連するIssueも自動でクローズ

実際の開発フロー (例)

1. Issue作成

gh issue create --title "ToDoの削除機能" -l enhancement

2. ブランチ作成

git checkout -b feature/delete-todo

3. 実装・コミット

git commit -m "feat: 削除ボタンを追加 #12"

4. Push・PR作成

git push origin feature/delete-todo
gh pr create --title "ToDoの削除機能" --body "Close #12"

5. レビュー・修正・マージ

gh pr merge 15

このプロジェクトで実践しよう

  • 課題を始める前にIssueを作成
  • 機能ごとにブランチを切る
  • 完成したらPRを作成
  • 自分でレビューしてからマージ

実際のチーム開発と同じフローを体験できます！
